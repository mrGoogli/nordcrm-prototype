Cosmos DB partition key strategy and composite index definitions. Correct partitioning is critical for performance at scale and for GDPR tenant isolation. All partition keys include tenantId to ensure data never crosses tenant boundaries at the storage level.
// Cosmos DB Containers & Partition Strategy
📦 tenants
Partition key: /id
Small, rarely written. Single partition sufficient. No cross-tenant contamination risk.
📦 users
Partition key: /tenantId
All queries scoped to a tenant. Low volume per tenant. Tenant isolation guaranteed at partition level.
📦 contacts
Partition key: /tenantId
Core CRM entity. Tenant-scoped queries dominant. GDPR erasure can target entire partition per tenant.
📦 accounts
Partition key: /tenantId
All account lookups are tenant-scoped. Stacc financial data co-located within tenant partition.
📦 deals
Partition key: /tenantId
Pipeline views always tenant-scoped. Cross-partition queries (by ownerId) handled by composite index.
📦 activities
Partition key: /tenantId
High-volume write container. Tenant isolation critical. TTL enabled per retention policy.
📦 tickets
Partition key: /tenantId
Support queries tenant-scoped. Complaint flag queries use composite index.
📦 ticket_messages
Partition key: /tenantId
High-volume. Field-encrypted bodies. TTL applied per ticket resolution + retention policy.
📦 campaigns
Partition key: /tenantId
Low-medium volume. All lookups tenant-scoped.
📦 lead_scores
Partition key: /tenantId
GDPR Art. 22 profiling records. Retain modelVersion for explainability. Separate from contacts for access control.
📦 rights_requests
Partition key: /tenantId
GDPR rights tracking. Low volume. DPO access only. No TTL — retain indefinitely for compliance evidence.
📦 audit_log
Partition key: /tenantId
DORA + GDPR immutable audit trail. No TTL. Append-only. High-volume writes. Azure Monitor backup for 1yr minimum. Separate dedicated throughput (RU/s).

Entity relationship map. In Cosmos DB (NoSQL), relationships are managed via embedded sub-objects or reference IDs — no foreign key constraints. References are validated at application layer. Cross-container queries use the Cosmos DB change feed or Azure AI Search for complex joins.
Core Relationships
tenants1 → NusersOne tenant has many CRM users (employees)
tenants1 → NcontactsStrict tenant isolation — contacts belong to exactly one tenant
tenants1 → NaccountsOrganisations owned by tenant
accounts1 → NcontactsOrganisation has many individual contacts (contactId → accountId)
accounts0..1 → NaccountsHierarchical: parent account → subsidiary accounts (parentAccountId)
users1 → NcontactsRelationship manager owns/is assigned to contacts
contactsembedstaccSnapshotStacc financial data embedded in contact document (field-encrypted)
contactsembedm365ProfileM365 Graph API metadata embedded in contact document
contactsembedgdpr objectGDPR consent & rights object embedded — always co-located with PII
Sales Relationships
contacts1 → NdealsContact can have multiple deals (one primary contact per deal)
accounts1 → NdealsOrganisation-level deals (accountId on deal)
users1 → NdealsSales rep owns deals (ownerId on deal)
deals1 → NactivitiesDeal timeline: all touchpoints linked via dealId
contacts1 → NactivitiesContact interaction history via contactId
activitiesrefm365 eventsm365EventId links to Microsoft Graph calendar/email (external ref)
Support Relationships

Marketing Relationships
