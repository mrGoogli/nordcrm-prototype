Overview & Authentication

All API requests must be authenticated via Microsoft Entra ID (Azure AD). The API uses OAuth 2.0 Bearer tokens issued by MSAL. Every request is scoped to a tenant, identified by the X-Tenant-Id header.

🔐 Authentication Flow
  • Protocol: OAuth 2.0 + PKCE (web/mobile)
  • Client Credentials: For server-to-server (M365, Stacc)
  • Token issuer: Azure Entra ID (MSAL)
  • Token lifetime: 1 hour access / 24h refresh
  • MFA: Enforced via Conditional Access
  • Scopes: crm.read, crm.write, crm.gdpr, crm.admin
🌐 API Design Principles
  • Style: RESTful, resource-oriented
  • Format: JSON (application/json)
  • Versioning: URL path (/v1/)
  • Pagination: Cursor-based (continuationToken)
  • Dates: ISO 8601 UTC always
  • IDs: UUID v4 strings
  • Money: Integer øre (1/100 DKK)
🏛️ GDPR API Obligations
  • Every write: Emits audit_log entry
  • PII reads: Log sensitiveAccess=true
  • Erasure: Cascades across all containers
  • Consent gate: Checked before any marketing write
  • Retention: TTL enforced via Cosmos DB policy
  • Data export: Machine-readable JSON (Art. 20)
🛡 DORA Obligations
  • Incident endpoint: POST /v1/incidents/dora
  • All 5xx errors: Trigger Azure Monitor alert
  • Circuit breaker: On Stacc + M365 integrations
  • Timeout: 30s hard limit on all requests
  • Health check: GET /v1/health (no auth required)
  • Audit log: Immutable, min 1yr retention

Rate Limits & Throttling

Rate limits are enforced at Azure API Management per tenant, per endpoint category. Financial and GDPR endpoints have stricter limits to prevent bulk data extraction (GDPR minimisation + DORA data security).

Endpoint CategoryLimitWindowNotes
Standard reads (GET)500 reqper minutePer tenant. Burst up to 50 req/sec.
Standard writes (POST/PUT/PATCH)200 reqper minuteIdempotency keys deduplicate within 24h.
DELETE operations50 reqper minuteSoft-delete preferred. Hard delete requires admin scope.
Bulk endpoints (/bulk/*)10 reqper minuteMax 500 records per bulk request.
Stacc financial data30 reqper minutePSD2 consent checked each call. Strict limit — financial data.
GDPR rights requests20 reqper hourAnti-abuse. Art. 12(5) allows refusal of excessive requests.
Audit log reads60 reqper minuteDPO/admin scope only. DORA audit access.
M365 sync trigger60 reqper hourGraph API underlying limit applies.
Webhook delivery (inbound)1000 reqper minuteStacc + M365 webhooks. HMAC-SHA256 verified.
Health check (/health)unlimited—No auth required. DORA uptime monitoring.
⚡ DORA Requirement All 429 rate limit hits are logged in Azure Monitor. A sustained rate of 429s from a single tenant triggers an automated ICT incident assessment per DORA Art. 17 — potential signs of a security incident or data extraction attempt.

Outbound Webhooks

The CRM publishes events to registered webhook endpoints. Used to integrate with external systems, Logic Apps, and custom automation. All webhook payloads are signed with HMAC-SHA256.

📤 Available Events
  • contact.created — New contact added
  • contact.updated — Contact fields changed
  • contact.consent_changed — Consent grant/withdraw
  • contact.erasure_completed — GDPR erasure done
  • deal.stage_changed — Pipeline stage transition
  • deal.won / deal.lost — Deal outcome
  • ticket.created — New support ticket
  • ticket.sla_breached — SLA deadline missed
  • ticket.complaint_filed — isComplaint=true
  • rights_request.received — New GDPR rights request
  • rights_request.overdue — 30-day deadline approaching
  • stacc.aml_flag — AML flag raised on contact
🔐 Webhook Security
  • Signing: HMAC-SHA256(secret, rawBody)
  • Header: X-CRM-Signature: sha256={hash}
  • Timestamp: X-CRM-Timestamp (reject if >5min old)
  • Retry: Exponential backoff, up to 72h
  • Timeout: 10s — respond immediately, process async
  • TLS: HTTPS only. TLS 1.2+ required on receiver.
  • IP allowlist: Available for tenant whitelisting
  • DORA: Delivery failures >15min trigger incident log
